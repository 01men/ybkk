# Feedback Loop Rules

> 这一份 Skill 对应文章里**最重要的设计原则**：
>
> **"Rule 越多，AI 的解释空间就越大。"**
>
> 唯一的解药是把"能判定的事"全部下沉成脚本，然后把"完成"的定义权从 AI 手里拿回来。

---

## 1. 软门禁 vs 硬门禁

| 类型 | 失败时 |
|---|---|
| **硬门禁** | 立即阻断流程，必须修 |
| **软门禁** | 不阻断，但在制品里留 WARN 痕迹 + 在审查报告里被显式提及 + 在交付报告"软门禁告警"段沉淀 |

**判断标准**：硬门禁宁少不多。如果一个东西不是**质量红线**，就别让它阻断。

示例（5 道门禁的硬/软配置）：

| 门禁 | 类型 | 理由 |
|---|---|---|
| 静态扫描（安全规则 + 拼写） | **硬门禁** | 安全性是红线 |
| 部署 staging 成功 | **硬门禁** | 跑不起来不算做完 |
| E2E 关键路径全过 | **硬门禁** | 关键路径失败 = 用户能立刻感知 |
| 覆盖率阈值 | **软门禁** | 高覆盖率是目标，但偶然 78% 不是世界末日 |
| 端到端 LCP < 3s | **软门禁** | 性能有波动 |

## 2. 基线对比（剥夺 AI 解释权）

### 为什么必须有基线

AI 跑完门禁后会汇报「通过 / 失败」，但 AI 对失败有**非常顽固**的解释能力：

> 「这个不是我引入的问题」
>
> 「这个警告之前就有」
>
> 「这里是历史 bug」

这些话每一句都可能是真的，但你不可能每次都翻 git log 去验证。

**基线对比的解药**：开发启动前先把所有门禁失败的状态冻结成基线，跑完做差集。

### 怎么拍基线

```bash
./scripts/gates/gate-baseline.sh --snapshot
```

→ 生成 `kb/gates/baseline.json`，记录每个门禁在「未开发」状态下的失败集合。

### 怎么比

```
fail_set_baseline = baseline.json[gate_name]  # 开发前的失败
fail_set_current  = run_gate(gate_name)        # 开发后的失败
new_failures      = fail_set_current - fail_set_baseline

if new_failures is empty:
    gate PASSED
else:
    gate FAILED with new_failures list
```

**只有「新增的失败」会阻断**。历史的、历史修不动的、用别的方法处理的 —— 都不在阻断范围内。

> 「这是历史问题」这把万能钥匙就被 AI 拿掉了。

### 基线什么时候更新

- **默认不要更新基线**。新需求不应该让「之前挂着的问题」消失。
- **以下情况**允许更新基线（人工触发，不是 AI 决定）：
  - 旧需求明确接受作为已知缺陷，并写进 `kb/artifacts/<req-id>/known-issues.md`
  - 旧问题被本需求彻底修好（那 `current - baseline` 也应该是 0）

**orchestrator 更新基线**前必须有人工关卡。

## 3. 软门禁失败也要留下「伤疤」

> "你不能阻止 AI 偷懒，但你可以让偷懒变得很醒目。"

软门禁失败时：

1. 在 `.gates-state.json` 里写一条 `warnings` 记录（含阶段、门禁、失败项）
2. 在审查 Agent 的产物 `06-review-report.md` 软门禁告警段被显式提到
3. 在 `07-delivery-report.md` 的「软门禁告警」段沉淀

**绕过这件事在视觉上变得很不舒服**——审查 Agent 必读到，交付报告必有黄色标记。

## 4. 熔断机制（防"无限打回"循环）

任何阶段连续打回 ≥ 3 次 → 自动触发熔断暂停，弹窗：

```
⚠️ 本次需求在「开发」阶段连续打回 3 次。可能不是单点错误，而是需求/方案本身需要重新对齐。
建议：
  1. 重读 01-requirement-doc.md，看是否需要拆分
  2. 重读 02-design-doc.md，看是否方案本身需要重做
  3. 或把需求切小后重开
```

熔断暂停后用户必须**人工解除**（输入「继续打回」/「重做方案」/「拆需求」）。

## 5. 任务清单 03-tasks.md 的「可验证性」

每一条 task 必须含：

- [ ] TASK-NNN: <描述>
  - **验证方式**（怎么判定完成）
  - 涉及文件 / 模块
  - 估算人时
  - 依赖（前置 task）

**反例**：

- ❌「实现登录功能」（怎么算完成？）
- ✅「实现登录功能。验证：能在 tests/auth.test.ts 看到 happy path + 1 个错误用例。」

## 6. 阻塞项文档的强制 schema

```markdown
# [BLOCKER] <类型> — <一句话>

- 类型: <范围溢出 / 验收缺失 / 质量 / 前端 / 安全 / 数据迁移 / 性能 / ...>
- 位置: <文件:行 或 模块>
- 现状: <代码在做什么>
- 期望: <应该是什么>
- 复核: <怎么验证已经修好>

## 影响半径
- 影响上下游: <是 / 否>
- 是否需要回滚: <是 / 否>
- 回滚方式: <具体动作>
```

存到 `kb/artifacts/<req-id>/blockers/`，**每一类问题独立文件**。

## 7. 体系演进日志 kb/changelog.md

**任何**对体系的修改都要写一条 changelog：

```markdown
## YYYY-MM-DD: <一句话标题>

- 改动: <改了哪个文件、为什么>
- 触发原因: <那个需求 / 那次撞墙>
- 影响面: <哪些 Skill / 哪些 Agent 会受影响>
- 回滚方法: <如果改坏了怎么改回去>
```

> 否则 3 个月后你想不起来「为什么当初这么定」。

## 8. 反馈循环的「副作用」

> 文章里有一句金句：**「Harness 不是为了让 AI 看起来更聪明，是为了让 AI 在复杂工程里更可控、更可靠、更可维护。」**

反馈循环的存在不是为了惩罚 AI，是为了**剥夺解释权**，让 AI 的「完成」从「我觉得」变成「脚本通过」。
