# Design Doc Template

方案 Agent 写 `02-design-doc.md` 时强制按这个模板。**漏字段视为方案不完整**。

## 模板

```yaml
# 02-design-doc.md — 技术方案
req-id: <REQ-XXX>
based-on: 01-requirement-doc.md
created: YYYY-MM-DD
author: solution-architect
version: 1

## 1. 方案摘要（200 字内）
（用 200 字说清：动哪些模块、用什么技术决策、风险点在哪）

## 2. 架构变更
- 新增模块: [...]
- 修改模块: [...]
- 数据模型变更: [...]
- 接口契约变更: [...]

## 3. 关键技术决策
- 决策 1: <方案描述>
  - 备选: <备选方案>
  - 取舍: <为什么这个>
- 决策 2: ...

## 4. 数据流
（请求怎么到响应，中间经过什么，**用文字 + 短序列图**）

## 5. 失败模式
- 场景 A: <怎么兜底>
- 场景 B: ...

## 6. 安全考量
- 鉴权: <如何>
- 数据脱敏: <如何>
- 限额: <如何>
- 审计日志: <如何>

## 7. 性能考量
- 预估 QPS: N
- 关键路径耗时: ≤ Mms
- 慢查询 / 慢接口: 列出

## 8. 测试策略
- 单测: 哪些类必须 ≥ 80%
- E2E: 哪些路径必须跑
- 安全扫描: 哪些必须过

## 9. 部署策略
- 灰度计划: <如何灰>
- 回滚计划: <一键回滚怎么做>
- 监控指标: <盯什么>

## 10. 范围（明确边界）
- 包含: <必做>
- 不包含: <明确不做>

## 11. 风险自评
| 风险类型 | 等级 | 兜底方案 |
|---|---|---|
| 范围溢出 | low/med/high | ... |
| 数据回退 | ... | ... |
| 接口破坏 | ... | ... |
```

## 反例（绝对不能这么写）

- ❌ §1 方案摘要写 800 字（用户 / orchestrator 没耐心看）
- ❌ §3 关键技术决策只列标题不写「取舍」
- ❌ §5 失败模式是空的（被 reviewer 必打回）
- ❌ §9 部署策略只写「部署到生产」（没说怎么灰度、怎么回滚）
- ❌ §10 没写不包含 —— reviewer 会打回

## 与需求文档的连接

- 02-design-doc.md **第一行必须包含** `based-on: 01-requirement-doc.md`
- 任何对 PRD 的偏离 → 在 §11 风险自评里加一行，并在制品目录 `blockers/01-design-blocked-by-req.md`

## 任务清单（03-tasks.md）必须在设计文档之后写

任务清单的格式见 `assets/skills/feedback-loop-rules/SKILL.md` §3。
