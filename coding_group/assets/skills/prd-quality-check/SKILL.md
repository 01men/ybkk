# PRD Quality Check

## 用途

任何写需求文档的 Agent **必跑**这一 Skill，对自己产出的需求理解打 5 维度评分。

## 5 维度评分（每维 0~100）

| 维度 | 评估什么 | ≥ 60 才合格 |
|---|---|---|
| **目标明确性** | 一句话目标能不能被不读 PRD 的人听懂？ | 一句话目标不为空，且无歧义 |
| **验收可验证性** | 每条验收标准能不能跑 E2E 验证？ | 每条都是「可观察、给定输入有唯一输出」 |
| **范围清晰性** | "不做什么"清单是否明确？ | 非目标段至少 3 条 |
| **数据/接口契约** | 数据来源、字段定义、上下游接口是否清楚？ | 输入输出字段有 schema 段 |
| **风险与依赖** | 外部依赖、性能、迁移风险是否识别？ | 风险段至少 2 条 |

**通过线**：每维 ≥ 60，**且**总分（5 维平均）≥ 60。

**未通过** → 哪一维 < 60 就针对性补问用户；最多 3 个问题（最重要的优先）。

## 打分方式（agent 自己填）

```markdown
## 7. PRD 完整性自评（5 维 0~100）
- 目标明确性: <数字>
- 验收可验证性: <数字>
- 范围清晰性: <数字>
- 数据/接口契约: <数字>
- 风险与依赖: <数字>
- 总分: <5 维平均>
```

## 反例（绝对不能这么写）

- ❌「目标明确性: 80」（不写依据）
- ❌ 5 个维度都给 80（看都没看）
- ❌ 一句话目标明显模糊却给 90
- ❌ 验收标准写「系统要友好」（不可验证）

## 通过的样例

```yaml
1. 一句话目标: 用户能在 https://example.com/p/<id> 路径上传一张 jpg，60s 内得到截图回显
2. 目标用户与场景:
   - 主要用户: 博客作者
   - 触发场景: 想给文章配一张网站截图但不想装本地工具
   - 用户想要: 复制 URL → 5s 内拿到图
3. 用户故事:
   - AS 用户 I WANT 粘贴 URL 并等待 5s SO THAT 拿到无水印截图
4. 验收标准:
   - [ ] 输入 https://example.com，5s 内返回非空 PNG
   - [ ] 输入非 URL 返回 400 且错误信息含 "invalid url"
   - [ ] 同一 URL 二次请求 5s 内命中缓存
5. 非目标:
   - 不做视频截图
   - 不做整页滚动截图（v1 只截可视区）
   - 不做登录用户的私有页面
6. 影响半径: modules=[frontend(1), backend(1), deploy(1)], files ≈ 8, 风险=low
7. PRD 自评:
   - 目标明确性: 90
   - 验收可验证性: 85
   - 范围清晰性: 90
   - 数据/接口契约: 70
   - 风险与依赖: 75
   - 总分: 82
```
