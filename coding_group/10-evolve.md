# 体系演化：跟着你的产品一起长大

> 这套方案不是一次性产物，它会跟着你长大的产品一起演化。这一章讲怎么演化、什么时候加什么、什么时候砍什么。

---

## 一、第 1 周：能跑通最小需求

目标：跑完 3~5 个真实需求，每个需求体感「确认 1~2 次就能收到产物」。

不要做：
- ❌ 加更多 Skill
- ❌ 加更多门禁
- ❌ 加更多人工关卡

要做的：
- ✅ 把每一棒跑出来的制品都看一眼，看有没有「缺字段」「逻辑有问题」的地方
- ✅ 把每一份 review 报告看一眼，看阻断项是否站得住
- ✅ 把每一份失败状态看一眼，看是不是「基线对比」逻辑在工作

如果发现「套路性失败」——比如某个 Skill 反复让 Agent 走错路——改 Skill。一个改完跑通再看下一个。

---

## 二、第 2-4 周：体系稳定

跑通 10+ 需求之后，你会感觉到：
- 周期可预测（30-75 分钟一个需求）
- 出错多在已知几类（基线配置错、E2E 偶发、PR reviewer 不响应）
- "完成"的定义不再被 AI 质疑

到这个阶段可以：
- 把门禁脚本挂到 CI（GitHub Actions / GitLab CI）
- 把 PR 模板用上（每次发布 PR 时自动填 gate 报告）
- 把每一次事故写进 `kb/incidents/`

这个阶段也开始出现「端到端耗时瓶颈」——通常是门禁脚本 / E2E。可以：
- 把 Coverage 和 Lint 留在本地（30 秒内），
- 把 E2E + Deploy Test 放在 CI（并行，不占 IDE 时间）
- 视觉回归加并发（Playwright sharding）

---

## 三、第 2-3 个月：从「单产品」到「多产品」

当你有第 2 个产品：
- 抽一份「产品模板」仓库，含本方案的脚手架
- 每个新仓库 `git init` + 拉模板 + 改工具栈基线即可
- 跨产品共享的通用 Skill 抽到公共仓库 `kb-common/`

---

## 四、什么时候加新东西

### 加 Skill 的触发条件

> Agent 在某一类任务上反复纠结，写了 3+ 次阻塞项 = 缺 Skill。

例：
- 反复提「这块代码用什么 ORM」 → 写 `coding-conventions` § ORM 选择
- 反复问「怎么校验 JWT」 → 写 `security-rules` § 鉴权细节
- 反复写错的「PR 模板」 → 写 `pr-template` Skill

### 加门禁的触发条件

> 同一个错误出现 2 次 = 该门禁化。

例：
- 两次把 API key 写进 git → 加 `lint:secret-in-code`
- 两次没跑迁移就部署 → 加 `deploy-test:check-migrations`
- 两次 model 字段没改 schema → 加 lint 规则

### 加 Agent 的触发条件（慎重）

> **Skill 和门禁不够用时再考虑加 Agent。**

加 Agent 的硬标准：
- 这一棒有显著的「独立认知工作」（不是脚本能干的）
- 这一棒有独立的失败模式
- 这一棒失败回退的位置不一样

否则就用 Skill 解决。

---

## 五、什么时候砍东西

> 加得越多维护成本越高。该砍就砍。

### 砍 Skill 的触发条件

- 半年没人调用
- 调用了但产出跟不调差不多

### 砍门禁的触发条件

- 假阳性 > 1/10 → 调阈值或转软门禁
- 阻断但 developer 必 "skip CI" 才能过 → 砍

### 砍 Agent 的触发条件

- 跑通 < 5 次的真实需求
- 跟另一个 Agent 责任重叠

## 六、长期维护的资源

| 资产 | 维护节奏 |
|---|---|
| `kb/tooling.md` | 工具栈变化时更新（季度复盘） |
| `kb/changelog.md` | 每次体系修改一行 |
| `kb/skills/` | 出错时改；改完跑基线 |
| `kb/gates/baseline.json` | 默认不更新；只在已知缺陷被接受时人工更新 |
| `kb/incidents/` | 每次事故一份；事故复盘 |
| `.gates-state.json` | **不进 git**，每次跑刷新 |

---

## 七、判断体系「健康」的硬指标

| 指标 | 健康值 | 异常 < | 异常行动 |
|---|---|---|---|
| 单需求平均耗时 | 30-75 分钟 | > 120 分钟 | 看 5 道门禁哪一道最慢 / 看哪一棒打回最多次 |
| 单需求平均人工按键次数 | 2~3 次 | > 5 次 | 看人工关卡设计 / 简化选项 |
| E2E 一次通过率 | ≥ 70% | < 50% | 看 E2E 是不是设计得太脆 / 加 retry |
| Review 阻塞率 | ~25% | > 50% | 看阻塞项是不是写得假大空 / 调阈值 |
| 软门禁告警被显式提及 | 100% | < 100% | orchestrator 实现漏了；修 |
| 体系知识改一处触发跑回归 | 100% | < 100% | 把 Skill 改动接到 CI smoke |

## 八、给 6 个月后的你留几句话

> 你现在可能觉得这套体系「规矩太多」「迭代慢」「节奏被绑死」。

6 个月后，你的直觉会变：

- 你的产品会有第一个付费用户，第一个事故，第一个被攻击的接口
- 那时候你会知道：当初「这条规则为什么要这么写」，现在回头看就是答案
- 这套东西不是「规矩」——它是**纪律**。是让 1 个人的产出像 5 个人的纪律
- 也是让你放心去睡觉、放心去陪家人、放心去度假的纪律

走窄门，慢就是快。

---

## 九、3 个永远不能省的事

```
1. 体系的知识沉淀（不是 Memory，是仓库里的 kb/）
2. 制品的强制 schema（不是 AI 自由发挥，是固定模板）
3. 反馈循环（不是"AI 觉得"完成，是"脚本判定"完成）
```

这 3 件你哪天砍了，那天体系就开始崩塌。

## 十、你想扩展到团队 / 公司时

本方案是「个人 + Mavis / Claude Code」的规模。
扩展到多人协作（5+ 人）需要：

| 现在 | 多人版 |
|---|---|
| 1 个仓库 | 1 个 monorepo + 多个产品仓库 |
| 1 个总控 | 1 个总控 + per-team orchestrator |
| 11 个 Skill | 25-30 个 Skill，每个业务一个 |
| 5 道门禁 | 7-10 道门禁 |
| 1 个人工关卡节奏 | 按团队节奏调整（建议每需求 4-6 次） |
| 1 个总控 Agent 由 Mavis | 公司自建 orchestrator + 共享的 GitHub App + Slack/钉钉 |

到那时候再看本方案的扩展版。或者——找我（Mavis）帮你演化。
