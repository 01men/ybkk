# 方案总览：四块拼图 + 六层积木

> 这一章回答一个问题：为什么不能更简单？比如只让一个 AI 干到底。

---

## 一句话总结

**AI 的瓶颈从来不是模型，是协作、流程、信任。** 本方案用"4 块拼图 + 6 层积木"把这件事工程化。

---

## 为什么不能"一个 AI 干到底"

很多人在这一步栽过：让 GPT 干"读 PRD、写方案、写代码、写测试、自审、提 MR"。
跑两个需求就开始出现一组症状：

| 症状 | 根因 |
|---|---|
| 上下文一团乱，分不清 PRD 写的 vs 它自己脑补的 | 单 turn 上下文爆炸，1M token 都不够 |
| 自己的方案自己审，几乎不会否决 | 模型对"自己的产出"天然没有否决欲 |
| 反复说服自己"问题不大，继续吧" | 倾向推进，不倾向停下 |
| 给你列了一堆任务清单，全勾完，但其实啥也没真做 | "完成"的口头定义权在它手上 |

第 4 行最致命：**当 AI 自己既是运动员又是裁判，"完成"就是它说了算。**

本方案要做的事，就是把"完成"的定义权从 AI 拿回来，分给"约束、反馈、知识、进化" 4 块拼图。

---

## 四块拼图（缺哪块都不行）

```
[ 约束与流程 ] ←——→ [ 反 馈 ]
      ↑                ↑
      │                │
[ 知 识 库 ] ←——→ [ 进  化 ]
```

| 拼图 | 它管什么 | 你这边对应的层次 |
|---|---|---|
| **约束与流程** | AI 按什么顺序、什么边界做事；谁接谁的棒 | Workflow + Agent + Skill + Rule |
| **反馈** | 做完以后系统能不能对结果判对错 | Scripts（门禁） + 基线对比 |
| **知识库** | 已经踩过的坑、已经定好的规约，谁都能查到 | Skill 文档 + `AGENTS.md` + `kb/` 目录 |
| **进化** | 新的坑出现后，怎么把它写回拼图里 | Runbook + 演进日志 |

缺约束 → AI 把架构边界冲掉，开发偷偷加范围外的东西
缺反馈 → "我改完了"全靠口头，代码审查通过后接口测试一跑就挂
缺知识 → AI 在新需求里把公共工具里已有方法又写一遍
缺进化 → 改 prompt 时不知道上一次为啥这么定，反复横跳

## 六层积木（每层只干一件具体事）

| 层次 | 一句话定位 | 在本方案里的形态 |
|---|---|---|
| **Rule** | 自然语言写的"家法" | 一份 `AGENTS.md`（仓库级） |
| **Skill** | 单类任务的标准操作手册 | 11 个 `SKILL.md` |
| **Sub Agent** | 专职角色，按职责分开 | 4 个 + 1 个总控 |
| **Workflow** | 谁接谁的棒、产物是什么 | 9 阶段状态机 |
| **Scripts** | 把"完成"变成机器可判定的 | 5 道门禁脚本 |
| **MCP** | 让 AI 在受控边界内动外部系统 | 3 个 MCP |

### Rule 与 Skill 的边界

**Rule**：写得短、写得模糊、对应"态度"
> "代码必须有注释"
> "不要瞎改 PRD 没要求的功能"

**Skill**：写得长、写得具体、对应"操作"
> "数据访问层必须用 ORM、必须 mock 单测、覆盖率 ≥ 80%、错误必须用统一错误码包"

> **判断标准**：能判定的（即"完成"可以由机器判断"成 / 不成"）一律下沉成 Skill 或 Scripts；不能判定的（比如"代码要优雅"）才留在 Rule。

### Agent 和 Skill 的边界

- **Skill** 是无状态的文档加载。它不要 turn，不要 token，不读对话历史。
- **Agent** 是带状态、带 turn、带 token 成本的。Agent 一旦存在就贵。

> **YAGNI**：能用 Skill 解决就别开 Agent。一上来就拆 7 个 Agent 一定跑不稳，因为协调成本远大于并行收益。

## 关键纪律（所有的设计都围着这几条）

### 纪律 1：下游不可改上游产物

需求 Agent 写了需求理解 → 方案 Agent 不能改它，只能提阻塞项由总控打回上游。
方案 Agent 写了设计 → 开发 Agent 不能改它。
开发 Agent 交了代码 → 审查 Agent 不能改代码，只能产阻塞项。

**这是责任边界的硬规矩，比任何 prompt 调优都关键。**

### 纪律 2：能写脚本就别让 AI 解释执行

"覆盖率 ≥ 80%" 是脚本判定的，不是 AI 评的。
"门禁是否通过" 是脚本判定的，不是 AI 评的。
"提交、推送、拼链接" 是 bash 做的，不是 AI 做的。

> Rule 越多，AI 的解释空间就越大。同一条规则放在不同需求里，AI 总能找到看上去合理的理由绕过去。脚本把"完成"的定义权拿回来。

### 纪律 3：基线对比剥夺 AI 的解释权

AI 跑完门禁后会汇报"通过 / 失败"，但 AI 对失败有非常顽强的解释能力："这是历史问题"、"不是我的范围"。

**做法**：开发启动前先把当前所有门禁的状态冻结成基线；跑完做差集：
- 失败集合 A（开发前）
- 失败集合 B（开发后）
- 新增的失败 = B − A
- 只有"新增的失败"会阻断。

### 纪律 4：半自动比全自动更稳

> "不能每分钟点一次以上"是我们后来定的硬约束。

理论上你可以让 AI Reviewer 去判断"这个方案能不能进开发"。我们试过——不行。模型对自己生成的东西天然没有否决欲。99% 的情况它会推进。

关键节点必须留人。半自动的体感比全自动更难调，但只有这样团队才不会绕过工具退回「Vibe Coding」。

### 纪律 5：项目级知识落到仓库

4 个工程师同时在不同需求上跑 Harness，如果项目级知识只存在某个人的 Memory 里，4 个人的 Memory 会互相矛盾。

**纪律**：团队要对齐的东西必须落到仓库；Memory 只放纯个人偏好（比如"回答用中文"）。

## 九阶段接力赛（一棒看完就能画出来）

```
需求理解 → 需求确认 → 技术方案 → 方案确认
    → 开发实施 → 自测验证 → 代码审查 → 验收 → 交付收尾
```

完整定义见 `06-workflow.md`。这里只讲"为什么是 9，不是 3 或 30"。

每一阶段的存在都对应一类历史教训，强行合并的下场是"代码审查过了 → 接口测试挂了 → 全白做"。反过来，每一阶段都要过同一道筛：

> 它是否有独立的失败模式？
> 它失败时回退的位置是否不一样？
>
> 两个都是"是" → 必须独立。否则就合。

## 第一天不要做的事

列下来当标语：

```
❌ 第一天就拆 5 个 Agent
❌ 让 AI 跑确定性脚本（提交、推送、拼链接、跑迁移——全部下沉成 bash）
❌ 让下游 Agent 改上游产物
❌ 把所有失败都设成硬门禁
❌ 没有基线对比就相信 AI 说的"这是历史问题"
❌ 把团队规约存进 AI 的个人 Memory（必须落到仓库）
❌ 关闭 Skill，只信 prompt
```
